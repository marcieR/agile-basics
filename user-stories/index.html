Story Writing

How to Write Great
User Stories

The format is simple. The discipline isn't. Here's how to write stories your team can actually build from.

📝

As a… I want… So that…

The format forces three things: who the user is, what they need, and why it matters. Skip any part and you lose context the team needs to make good decisions.

A good example
Good story

As a project manager, I want to filter tasks by assignee, so that I can see each team member's workload at a glance.

Breaking it down
  • As a [persona]
    Defines who benefits Prevents building for imaginary users. A named persona forces you to think about the real person and their real context — not a generic abstraction.
  • I want [action]
    The capability needed, not the solution Focus on what the user needs to do, not how it should be implemented. "Filter by assignee" leaves the design open — "add a dropdown filter" locks it in prematurely.
  • So that [outcome]
    The business value Keeps the team focused on impact, not just output. When the implementation gets complex, this is the north star that guides trade-off decisions.
A bad example — and why it fails
Bad story

As a user, I want the system to be better.

"User" is not a persona — it gives no context about who this is or what they care about. "Be better" is not an action — it's a vague sentiment that can't be designed, estimated, or tested. And without a "so that", there's no measurable outcome to aim for. Every part fails.
💡

The INVEST Test

Every story should pass all six criteria before entering refinement.

I

Independent

Stories shouldn't depend on each other. Dependencies create sequencing problems in planning and make it impossible to prioritise freely.

N

Negotiable

The story isn't a contract. Details can change through conversation — only the goal is fixed. Stories are placeholders for a conversation, not specifications.

V

Valuable

Delivers value to a real user or the business. If you can't answer "who benefits?", the story isn't ready. Technical tasks that have no user-facing impact should be framed differently.

E

Estimable

The team can estimate it. If they can't, it's either too big, too vague, or needs a spike to reduce uncertainty before planning.

S

Small

Completable in a single sprint. If it can't fit, split it. Large stories always slip — they carry hidden complexity and make sprint commitments unreliable.

T

Testable

Has acceptance criteria that can pass or fail. Untestable stories ship ambiguity. If QA can't write test cases from the AC, the story isn't done being written.

✅

Acceptance Criteria That Actually Work

AC is the contract between the PO and the team. Good AC is specific, testable, and complete — covering happy paths, error states, and edge cases.

Given / When / Then — Gherkin format
  • Given the user is on the task list,
    When they click the filter dropdown,
    Then they see a list of all assignees.
  • Given no tasks match the filter,
    When the filter is applied,
    Then an empty state message is shown.
Rules-based format
  • Filter shows only assignees with active tasks
  • Selecting multiple assignees uses OR logic
  • Filter state persists on page refresh
  • Mobile: filter collapses into a full-screen sheet
Both formats work. Pick one and use it consistently. The goal is that QA can write test cases from AC without asking questions. If they need to ask, the AC isn't done yet.
✂️

How to Split a Story That's Too Big

The most common mistake is treating a large story as a planning problem. It's a writing problem. Every large story can be split — here's how.

  • 1
    By workflow step Split along the natural steps of a user flow. "Submit a form" becomes "enter data", "validate data", and "confirm submission" — each independently shippable.
  • 2
    By user role If different users do different things, split by persona. What an admin can do and what a standard user can do are often two distinct stories.
  • 3
    By data variation Handle one data type first, then others. "Import CSV files" before "import Excel files" — same feature, smaller increments, faster feedback.
  • 4
    By happy path first Build the success case, then add error handling and edge cases as separate stories. The happy path is almost always independently valuable.
  • 5
    By interface Deliver the core functionality first — API or backend — then build the UI on top. Backend-first stories give the team an integration point without blocking on design.
  • 6
    By acceptance criteria If a story has 8 AC items, it's probably 2–3 stories. Split the AC, not just the points. Each chunk should have a coherent, independently testable outcome.
